## AI Booking Assistant

A backend-controlled conversational assistant built into the salon/barbershop booking platform. Customers ask natural-language questions about services, masters, and available slots. The assistant understands the request, validates it against real backend data, and guides the user into the normal booking flow.

**Core principle:**

> AI understands. Backend verifies. User chooses. Existing booking/payment flow confirms.

The AI layer is read-only. No booking is created, no payment is initiated, and no business rules are bypassed inside the assistant. All booking, payment, and admin workflows go through the standard backend endpoints.

---

### What the assistant can do

- Answer questions about active public services (list, price, duration, description)
- Answer questions about active public masters (list, bio, capabilities)
- Extract booking intent from natural language (service, master, date, time)
- Ask for missing details when a request is incomplete
- Check exact slot availability through the backend availability engine
- Search flexible availability across date ranges and weekday combinations
- Handle typo-tolerant weekday input (e.g. `mondau` → Monday)
- Return real available slots from the database
- Return prefill buttons that open the booking form pre-populated with service, master, date, and time
- Open the standard booking form with `book manually` or `open booking form`
- Reset the conversation state with `start over`
- Fall back safely to the booking form if the AI provider is unavailable

---

### What the assistant cannot do

- Create a booking directly
- Create a Stripe PaymentIntent
- Confirm or process payment
- Issue refunds
- Access admin-only or private data
- Modify master schedules
- Mutate the database directly
- Bypass backend availability validation
- Promise a slot is available without a backend check

### Architecture

```text
backend/app/ai/
  client.py              # Provider abstraction and daily rate limit
  prompts.py             # Structured extraction prompt builder
  schemas.py             # ExtractedBookingIntent and related Pydantic models
  providers/
    common.py            # Shared retry/error helpers
    gemini_provider.py   # Google Gemini provider
    groq_provider.py     # Groq provider

backend/app/modules/booking_ai/
  router.py              # POST /api/ai/booking-intent
  service.py             # Orchestration: local shortcuts → AI → response builder
  draft.py               # Booking draft merge, field validation, next-action logic
  schemas.py             # BookingIntentResponse and action types
  service_matching.py    # Fuzzy public service name matching and resolution
  master_info.py         # Master list/detail response builder
  availability_actions.py  # Exact availability check (read-only)
  flexible_availability.py # Flexible date/weekday/time range search (read-only)
  followups.py           # Command shortcuts: show more, start over, book manually
  response_messages.py   # Backend-generated factual assistant messages
  response_actions.py    # Prefill action builder
```

### Request flow

```text
Customer message
  → React AI chat widget
  → POST /api/ai/booking-intent
  → Local shortcuts checked first (start over, show more, greetings)
  → AI provider extracts structured intent (intent, service, master, date, time, weekdays)
  → Backend resolves real service and master records
  → Backend checks real availability if needed (read-only)
  → Backend generates factual assistant message from real data
  → Backend returns assistant_message + typed action list
  → Frontend renders message and action buttons
  → User clicks prefill button or opens booking form
  → Normal booking → Stripe → payment confirmation flow
```

The AI provider output is used only for structured intent fields (intent type, service query, master query, date, time, weekdays). The assistant message shown to the customer is always generated by the backend from verified data, never passed through from the AI provider.

### Environment setup

```env
# AI is optional. Booking system works normally if AI is unavailable.
AI_ENABLED=true

# Provider: gemini | groq
AI_PROVIDER=groq

# Gemini provider (Google AI Studio)
GEMINI_API_KEY=
GEMINI_MODEL=gemini-2.5-flash

# Groq provider (GroqCloud)
GROQ_API_KEY=
GROQ_MODEL=llama-3.1-8b-instant

# Optional safety limit for demo deployments
AI_DAILY_REQUEST_LIMIT=50
```

Configure only the provider you intend to use. The other provider's variables can be left empty. Never expose AI API keys in the frontend root `.env`.

---

### Fallback behavior

If the AI provider is unavailable (disabled, quota exhausted, network error, or invalid response), the endpoint returns a safe fallback message and a direct link to the booking form:

```text
AI assistant is temporarily unavailable right now, but the booking system is still
working normally. You can continue by using the booking form.
```

The core booking system — services, availability, Stripe payment, confirmation — is unaffected by AI provider availability.

### Design principle

The AI assistant is a safe language-understanding layer, not the business system.

| Layer | Responsibility |
|---|---|
| AI layer | Language understanding and intent extraction |
| Backend | Service/master data validation and business rules |
| Availability engine | Real slot calculation from schedules and confirmed bookings |
| Stripe | Payment authorization and confirmation |
| Database | Source-of-truth state for all entities |

The backend remains the authoritative source for services, masters, availability, bookings, payments, and admin workflows. The AI assistant cannot alter that state.
